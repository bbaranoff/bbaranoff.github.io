Rappels réseaux : notations

Réseaux IPv4 / IPv6 :

Une adresse réseau est définie sur l’ensemble des bits disponibles, soit 32 bits en IPv4 et 128 bits en IPv6.

Notation adresse IPv4 : 4 octets exprimés en base décimale séparés par des points.

Exemple : 10.0.1.123

Notation adresse IPv6 : 16 octets exprimés en base hexadécimale séparés par le caractère “deux points“ et suivie du masque de sous réseau en notation CIDR, les octets à 0 pouvant être omis.

Exemple : dead::beef/128

  • CIDR : Classless Inter-Domain Routing

Rappels réseaux : classes

Une partie des bits définit l’adresse du réseau (net id), et le reste l’adresse de l’hôte (host id)

ex. d’adresse réseau IPv4 :192.168.0.0

où le net id vaut 192.168.0 et le host id peut varier de 1 à 254 avec 2 valeurs particulières pour le host id :

  • tous les bits à 0 => adresse du réseau
  • tous les bits à 1 => adresse de broadcast (diffusion générale). Le masque de sous-réseau doit correspondre au découpage net id / host id afin que le résultat d’une opération ET logique entre adresse d’hôte et le masque de sous-réseau donne le net id.

ex. de masque de réseau IPv4 : 255.255.255.0 ou en notation CIDR : /24

Note : En IPv6, le masque de sous-réseau est systématiquement exprimé en notation CIDR

Rappels réseaux : classes

Historiquement IPv4, a défini des classes de réseaux définissant le nombre de bits alloués au net id et au host id.

Classe A : 1 seul octet pour le réseau (dont le premier bit vaut 0) / 3 octets pour identifier les hôtes.

=> adresses 0.0.0.0 à 127.255.255.255, soient 127 réseaux 
de 16 777 214 machines.

Classe B : 2 octets pour le réseau (dont les 2 premiers bit valent 10) et 2 octets pour identifier les hôtes.

=> 128.0.0.0 / 191.255.255.255, soient 16383
réseaux de 65 534 machines.

Classe C : 3 octets pour le réseau (dont les 3 premiers bit valent 110) et 1 octet pour identifier les hôtes.

=>192.0.0.0 / 223.255.255.255, soient 8191
réseaux de 254 machines.

Classe D : Les 4 octets sont utilisés pour définir l’adresse de réseau (dont les 4 premiers bit valent 1110)

=> adresses de multicast (ou multidiffusion : émission des paquets vers un groupe de récepteurs).

=>224.0.0.0 / 239.255.255.255, soient 268 435 455
groupes.

Classe E : Les 4 octets sont utilisés pour définir l’adresse de réseau (dont les 4 premiers bit valent 1111)

=>240.0.0.0 / 225.255.255.255, soient 251 658 240
réseaux (réservés par l’IANA).

A partir de 1990, en raison de la pénurie d’adresses IPv4, le routage internet fait appel à la notion de sous-réseaux, dans laquelle un host id est découpé en adresse de réseau / adresse de sous-réseau avec un masque de sous-réseau adapté en conséquence.

ex. : 172.16.1.23 / 24

La conséquence immédiate du découpage en sous-réseaux, est la nécessité de définir de nouveaux protocoles de routage car les protocoles standards de l’époque sont tous basés sur le routage des classes (classfull), utilisant le masque réseau implicite à l’adresse IP à router. C’est à partir de 1993 que la notation CIDR apparaît avec la mise en œuvre de nouveaux protocoles de routage (BGPv4, OSPF, EIGRP, RIPv2) adaptés au routage des sous-réseaux sur internet.

À noter qu’IPv6 paraît sans cette notion de classes de réseaux, mettant uniquement en œuvre la notion de sous-réseaux.

Périphériques réseau virtuels

Le noyau Linux permet également de créer des périphériques virtuels qui seront utilisés pour communiquer programmatiquement (VPN, réseaux virtuel, etc …). Linux supporte 2 types d’interfaces virtuelles :

  • TUN (pour TUNnel réseau) qui simule un périphérique réseau activé en couche 3 du modèle
  • OSI (couche réseau).
  • TAP (pour NeTwork Access Point) qui simule un périphérique virtuel en couche 2 du modèle OSI (couche liaison). Ces interfaces virtuelles sont généralement automatiquement créées par un programme mais peuvent également l’être par des commandes d’administration :
# tunctl -n -t tun0
# tunctl -p -t tap12

Ou encore avec la commande ip :

# ip tuntap add mode tun tun0
# ip tuntap add mode tap tap12

Périphériques ponts réseau Le noyau Linux permet de configurer des ponts réseau qui vont alors opérer comme un commutateur physique auquels il il faudra rattacher les interfaces réseau (physiques ou virtuelles) de la machine. Les nom des interfaces ponts sont laissés à la discrétion de l’administrateur, mais généralement nommés [X]br[X] selon l’utilisation qui en sera faite (par exemple br0 pour un pont agrégeant des périphériques physiques ou des interfaces reliées à un VPN, mais ce pourrait être virtbr0 pour un pont mettant en œuvre des connexions réseau vers des machines virtuelles).

Lorsqu’un pont est configuré, c’est celui-ci qui porte l’adresse IP et non pas les périphériques qui y sont intégrés.

Protocoles ARP / NPD

Les protocoles ARP (Adress Resolution Protocol) et NDP (Neighbor Discovery Protocol) pour l’IPv6 permettent de traduire une adresse de la couche 2 OSI (couche liaison et donc typiquement une adresse MAC) en une adresse de la couche 3 OSI (couche transport et donc typiquement une adresse IP). Le noyau linux maintient un cache ARP / NDP lequel peut être administré par la vieille commande arp ou bien par la commande ip (sous-commande neigh).

$ ip neigh show
192.168.66.253 dev br0 lladdr 00:24:a5:00:e8:40 REACHABLE
192.168.66.101 dev br0 lladdr 1c:83:41:28:01:39 REACHABLE

Protocoles de la couche réseau

La pile réseau du noyau Linux supporte divers protocoles de la couche de niveau 3 OSI allant des protocoles commutés tels que le vénérable X25 au célèbre protocole IP qui s’est totalement imposé. Les nombreux paramètres du protocole IP peuvent être consultés via le pseudo-répertoire /proc/sys/net/ mais rares sont ceux qu’il est intéressant de modifier … Juste au dessus du protocole IP – mais toujours en couche réseau 3 – les protocoles ICMP et IGMP vont fournir des outils de diagnostic concernant l’acheminement des paquets. Le plus célèbre étant implémenté par la commande ping. A ce même niveau, on peut également citer des protocoles de routage dynamique (par ex. OSPF).

Protocoles de la couche transport :

Au dessus des protocoles de couche réseau, la suite protocolaire TCP/IP permet de choisir entre un protocole en mode connecté : TCP (pour Transmission Control Protocol) ou bien un protocole plus simple, sans établissement de connexion : UDP (pour User Datagram Protocol).

C’est l’application (ou le protocole applicatif utlisé) qui va déterminer l’emploi de l’un ou l’autre de ces protocoles : si la perte de paquets n’est pas rédhibitoire, UDP sera préféré ; sinon l’utilisation de TCP sera nécessaire. A ce niveau de la pile ISO, TCP et UDP ne sont pas les seuls protocoles disponibles. Des protocoles spécialisés ont été définis plus tardivement, par exemple : le protocole RTP (Real Time Protocol) pour des applications nécessitant de gérer des contraintes temporelles strictes ou SCTP (Stream Control Transmission Protocol) qui reste similaire à TCP mais basé sur la transmission de trames plutôt que de suites d’octets et qui présente la particularité de connecter plusieurs adresses IP simultanément.

Routage statique :

Le routage consiste à permettre à la couche réseau de déterminer vers quelles interfaces diriger les paquets afin qu’ils arrivent à destination. Le noyau Linux maintient des tables de routage qui peuvent être configurées grâce aux commandes route ou ip route.

Exemples :

# route add -net 10.0.0.0 netmask 255.0.0.0 eth0
# route add -host 192.168.31.10 gw 192.168.1.1 eth0
# route del -net 10.0.0.0 netmask 255.0.0.0 eth0
# ip route add 10.0.0.0/8 dev eth0
# ip route add 192.168.31.10 via 192.168.1.1 dev eth0
# ip route delete 10.0.0.0/8 dev eth0

Routage par défaut

La route par défaut définit par quelle machine doivent transiter les paquets pour lesquels il n’existe pas de route définie : la passerelle par défaut.

Une machine chargée de faire transiter des paquets pour d’autres machines est également appelée passerelle. Sur une machine qui doit faire office de passerelle, le routage des paquets IPv4 doit être activé en positionnant le paramètre noyau ip_forward à 1, soit interactivement par la commande :

echo 1 > /proc/sys/net/ipv4/ip_forward

ou, pour rendre le comportement permanent, en modifiant ce même paramètre dans le fichier /etc/systcl.conf.

Passerelle IPv4 : masquerading / NAT Lorsqu’une passerelle doit faire passer des paquets depuis un réseau local vers le réseau global, la passerelle doit procéder à un traitement particulier du paquet transmis : elle doit remplacer l’IP source dans le paquet par sa propre adresse IP, et mémoriser cette opération afin de pouvoir transférer les paquets en réponse. Cette opération se nomme la translation d’adresse (Network Address Translation) et la technique mise en œuvre par le noyau Linux est le masquerading. Elle implique que tout paquet issu du réseau local à destination de l’Internet et vu comme provenant d’une adresse réseau unique : celle de la passerelle par défaut.

La commande iptables permet de mettre en œuvre cette technique :

# iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

Comme à l’accoutumée avec Debian, les empaqueteurs permettent de scinder les fichiers imposants en plusieurs fichiers plus simples. Dans le cas de la configuration du réseau, le fichier principal peut être découpé en différentes parties qui se trouveront alors dans le répertoire /etc/network/interfaces.d/

Services réseaux : sockets

Un service réseau sous GNU/Linux consiste en un programme qui “ouvre“ un “socket“ et l’utilise pour lire ou écrire des données sur l’interface réseau utilisée. Ce socket est en fait une abstraction de fichier avec lequel le programme va communiquer via les appels systèmes standards (fopen, fread, fwrite, etc …) et qui se charge d’acheminer les données vers le module noyau chargé de gérer le matériel sous-jacent.

C’est à la création du socket que le service définit les caractéristiques de la communication : socket réseau ou socket UNIX, protocole utilisé, port, etc.

Cette notion de port (définie au niveau de la couche transport OSI) permet de multiplexer les communications sur un périphérique déterminé. Un seul socket peut réserver sur un hôte les mêmes couples protocole / port. L’étendue des ports de communication (0 à 65535) a été standardisée : De 0 à 1023 (well-known ports) : attribués aux services réseaux les plus courants, ne peuvent être utilisés que par des programmes s’éxécutant avec les droits root. De 1024 à 49151 (registered ports) : sont assignés par l’IANA pour les besoins de services particuliers. De 49152 à 65535 (hight ports) : peuvent être utilisés de façon dynamique pour tout types de requêtes autres que celles précédemment définies.

Services réseaux

Historiquement, UNIX disposait d’un service réseau central : inetd (un super-daemon qui écoutait l’ensemble du réseau et se chargeait de lancer le programme configuré pour un couple protocole/port donné). Des scories d’inetd demeurent d’ailleurs sur les systèmes modernes sous la forme de deux fichiers de configuration :

  • /etc/hosts.deny : permet d’interdire la communication pour un couple service:IP (par ex. la ligne sshd: 192.168.1.250 interdirait la connexion SSH depuis cette IP)
  • /etc/hosts.allow : tout l’inverse du précédent (par ex. la ligne ALL: 92.168.1.250 autorise toute communication y compris s’il y a des restrictions dans /etc/hosts.deny)

Actuellement, les services réseaux sont plutôt implémentés sous forme de daemons autonomes qui vont se chargent directement d’initialiser et opérer sur les sockets réseaux ; et la majorité d’entre-eux vont malgré tout respecter le fonctionnement du vénérable inetd via les deux fichiers de configuration précédents (à vérifier dans le man toutefois).

Résolution de noms

La mécanique de résolution de nom des hôtes réseau permet de pouvoir cibler un ordinateur sur le réseau de manière symbolique, et non plus avec sa seule adresse IP. Les systèmes UNIX permettent de paramétrer le fonctionnement de cette mécanique via le fichier historique /etc/host.conf, dont le contenu comporte quelques directives du type : order hosts,bind multi on La directive order détermine l’ordre des méthodes appliquées lors d’une résolution. Dans l’exemple précédent, le resolver vérifiera d’abord si le nom recherché figure dans le dans le fichier /etc/hosts et sinon lancera une requête DNS. A noter qu’il existe bien d’autres méthodes de résolution des noms que hosts et bind, ainsi nis, ldap sont également des méthodes possibles. La directive multi définit quant à elle si la résolution peut retourner plusieurs possibilités pour un nom donné ou doit simplement s’arrêter au premier nom récupéré.

Résolution locale de noms

Une machine Unix peut donc utiliser la résolution locale de noms qui se configure le plus simplement du monde à travers le fichier /etc/hosts. Ce fichier contient une liste de couples IP/noms canoniques séparés par des caractères blancs (espaces ou tabulations). Par exemple : 192.168.1.10poste1 poste1.admisys.lan La ligne précédente affecte l’IP aux noms de machines poste1 ou à son FQDN poste1.admisys.lan

Résolution de noms : DNS

Le protocole DNS (Domain Name Server) est le protocole internet permettant de résoudre les noms canoniques des hôtes réseau en adresses IP. Il s’agit d’une organisation hiérarchique constituée au premier niveau des TLD (Top Level Domain) gérés par des entités commerciales ou non (ex. l’IANA pour le TLD .org, ou l’AFNIC pour le TLS .fr) L’enregistrement d’un nom de domaine s’effectue auprès d’un registrar qui détient une délégation pour un ou plusieurs TLD, et enregistre les données de l’acquéreur du domaine auprès d’un registre particulier : la base WHOIS. Sous Unix, la configuration du DNS à utiliser s’effectue dans le fichier /etc/resolv.conf dans lequel on va – entre autres – renseigner des entrées nameserver qui vont pointer vers l’adresse IP d’un DNS à interroger afin d’effectuer la résolution du nom. Par exemple :

nameserver 192.168.1.254
nameserver 196.12.125.3

Procédure de dépannage

1- Vérifier l’application de la configuration réseau sur le périphérique (ip address show)

Si non OK, corriger et redémarrer le système Vérifier la résolution DNS (host [www.google.com](http://www.google.com/))

Si non OK, corriger dans /etc/resolv.conf et re-tester

2- Vérifier la connectivité avec la passerelle par défaut (ping gw)

Si non OK, vérifier les routes statiques Vérifier la connectivité avec l’hôte distant (ping [www.google.fr](http://www.google.fr/)) Si non OK, vérifier la passerelle (route par défaut, connexion Wan, etc.) et dans le cas d’une passerelle GNU/LINUX, et de passage d’un réseau privé vers un réseau public la mise en œuvre de la mécanique de MASQUERADE sur la passerelle. Vérifier l’éventuel pare-feu pour le service à joindre

Protocoles réseau

Un protocole réseau est un protocole de communication mis en œuvre sur un réseau informatique. Il est possible que plusieurs protocoles réseau forment des couches de protocoles.

Couches :

Afin de s’y retrouver au milieu des protocoles, et même de pouvoir en changer, on peut les hiérarchiser : par exemple les quatre couches du modèle TCP/IP. D’autres modèles plus complexes, comme le SNA d’IBM ou le modèle de l’OSI, comptant chacun sept couches, ont rencontré moins de succès pratique et sont plutôt évoqués aujourd’hui à des fins de complétude de la théorie. Considérée dans son ensemble, une suite de protocoles entre diverses couches forme ce qu’on nomme une pile de protocoles. Les termes « protocole » et « pile de protocoles » désignent également les logiciels qui implémentent un protocole. Les protocoles les plus récents sont standardisés par l’IETF dans le cas des communications sur Internet, et par l’IEEE ou l’ISO pour les autres types de communication. L’UIT-T prend en charge les protocoles et les formats des télécommunications. Dans le milieu du P2P, on considère l’ensemble des algorithmes d’un réseau comme son protocole. Protocoles réseau applicatifs Ce sont des protocoles qui sont classés dans la couche la plus haute du modèle OSI (couche 7 - application) et utilisent les standards définis par les protocoles de couches plus basses. Par exemple, HTTP se base sur la pile de protocoles suivants :

  • HTML (couche 6 - présentation)
  • SOCKS (couche 5 - session)
  • TCP (couche 4 - transport)

La plupart de ces protocoles applicatifs historiques définissent des messages textuels échangés en clair sur le réseau (Telnet, FTP, HTTP, SMTP, POP, IMAP, …) et peuvent donc être testés ou simulés avec une simple connexion Telnet. L’évolution de ces protocoles, pour des besoins de confidentialité et de sécurité, se sont vus ajoutés une surcouche de cryptage (généralement via le protocole TLS) qui va englober l’ensemble des messages textuels du protocole historique dans une communication cryptée. Enfin, de nouveaux protocoles ont été imaginés dans le but de remplacer des protocoles historiques peu sûrs par un équivalent sécurisé, par exemple : Telnet est remplacé par SSH.

Protocole réseau : DNS

Le protocole DNS (Domain Name System) est un service informatique distribué qui permet de traduire un nom d’hôte (FQDN : Full Qualified Domain Name) en adresse IP. Au tout début d’Internet, seule une liste de correspondances nom d’hôte → adresse IP était maintenue et mise à disposition publique par l’IANA sous forme d’un fichier HOSTS.TXT, ce mécanisme trop formel manquait de souplesse a été remplacée par le système DNS à compter de 1983. Outre cette fonctionnalité essentielle au fonctionnement du réseau des réseaux, le DNS permet de servir également d’autres types d’enregistrements pouvant intéresser d’autres briques de base d’internet : par exemple l’enregistrement de type MX est nécessaire au fonctionnement du protocole SMTP, en renseignant quel est le serveur chargé d’acheminer les courriels pour un domaine donné.Les domaines ciblés par le DNS consiste en une organisation hiérarchisée dont le sommet est appelé la racine et représenté par le symbole point (.). A chaque niveau, tous les sous-domaines doivent faire l’objet d’une délégation qui prend la forme d’une déclaration dans les enregistrement du domaine supérieur sous forme d’un enregistrement de type NS. Ces délégations créent des zones DNS qui comprennent les domaines délégués et leurs sous- domaines non-délégués.

Les domaines de premier niveau sont appelés TLD (Top Level Domain) : on y distingues les domaines génériques (gTLD) tels que .org, .net, .com, etc. et les domaines nationaux (ccTLD) Ainsi, la racine du DNS est gérée par un organisme indépendant : l’ICANN qui a été substituées à l’IANA, alors que chaque domaine national se voit géré par une agence ou un organisme dépendant de l’état (par exemple, c’est l’AFNIC qui gère les domaines en .fr). Au final, chaque gestionnaire de domaine peut déléguer la vente des domaines dont elle a la responsabilité à des entreprises tierces nommées REGISTRARS (Verisign, Gandi, OVH, etc..).

Les serveurs racine (ROOT SERVERS) sont donc maintenus par l’ICANN et contiennent les déclarations pour l’ensemble des DNS des TLD. Ils sont actuellement au nombre de 13, nommés [A-M].ROOT-SERVERS.NET et répondent à des adresses IP fixes qui sont directement renseignées au niveau de chaque serveur DNS déployé, afin de leur permettre de joindre les points d’entrée de toute requête récursive. (cf. fichier /usr/share/dns/root.hints fourni par le paquetlogiciel bind9 sous Debian 11) Un TLD particulier .arpa est introduit afin de permettre la résolution inverse (recherche d’un nom d’hôte à partir d’une adresse IP) qui consiste à tenter de résoudre un nom canonique dépendante du pseudo-domaine .in-addr.arpa. Par exemple, l’interrogation de l’IP 91.198.174.2, sera résolue via le nom canonique 2.174.198.91.in- addr.arpa.

Le DNS d’un domaine peut être interprété comme un registre fournissant des informations nécessaires à l’intégration du domaine au réseau (via le mécanisme de résolution des noms canoniques) mais également en mettant à disposition des informations supplémentaires sur lesquelles vont s’appuyer des services connexes. Ces différentes informations sont enregistrées dans le DNS sous forme d’enregistrements de ressources (Resource Record) de différents types. Un certain nombre de ces types d’enregistrements sont nécessaires au bon fonctionnement de la zone DNS (SOA, NS, AA, AAAA) alors que d’autres sont optionnels (PTR, MX, TXT, etc.). De façon générale, ces enregistrements sont configurés sous forme de lignes de texte répondant à un formalisme bien particulier, comprenant les champs suivants : [nom] [TTL] [CLASSE] [TYPE] [VALEUR]

Protocole réseau : DNS

Enregistrement SOA (Start Of Authority), par exemple :

[wikipedia.org](http://wikipedia.org/). IN SOA [ns0.wikimedia.org](http://ns0.wikimedia.org/).
[hostmaster.wikimedia.org](http://hostmaster.wikimedia.org/). 2010060311 43200 7200 1209600 3600
- DNS principal
- Responsable de la zone (e-mail @ remplacé par .)
- Serial (conventionnellement YYYYMMDDNN)
- Refresh (délai en secondes d’interrogation pour les DNS esclaves)
- Retry (délai en secondes avant nouvelleinterrogation en cas d’échec pour les DNS esclaves)
- Expire (délai en secondes d’expiration en cas d’impossibilité de mise à jour des esclaves)
- Negative TTL (délai en secondes de conservation dans les caches des DNS récursifs)
Une seule occurrence de cet enregistrement doit obligatoirement doit exister dans la zone DNS.

Enregistrement NS (Name Server), par exemple :

wikipedia NS [ns1.wikimedia.org](http://ns1.wikimedia.org/). Ces enregistrements définissent les différents serveurs de noms délégataires pour la zone. Tous les serveurs renseignés doivent faire autorité pour le domaine, car peu importe l’ordre de déclaration, le client est censé pouvoir interroger indifféremment chacun de ceux-ci. D’une manière générale, pour assurer la redondance du service, il est de bonne pratique de déclarer au moins 2 DNS situés sur des réseaux distincts.

Enregistrements permettent de mettre en place le mécanisme de résolution de noms ou de résolution inverse pour une zone :

Enregistrement A (Address Record), par exemple :

[test.wikimedia.org](http://test.wikimedia.org/). IN A 91.198.174.232

Enregistrement AAAA (Address Record IPv6), par exemple :

[test.wikimedia.org](http://test.wikimedia.org/). IN A 1002:0161:0024:0022::0c10:0068

Enregistrement CNAME (Canonical Name), par exemple :

 [test.wikimedia.org](http://test.wikimedia.org/). IN CNAME [www.wikimedia.org](http://www.wikimedia.org/).

Enregistrement PTR (Pointer Record), par exemple :

232.174.198.91.in-addr.arpa. IN PTR [test.wikimedia.org](http://test.wikimedia.org/).

Protocole réseau : DNS Enregistrement MX (Mail Exchanger), par exemple :

[wikimedia.org](http://wikimedia.org/). IN MX 10 [mchenry.wikimedia.org](http://mchenry.wikimedia.org/).
[wikimedia.org](http://wikimedia.org/). IN MX 50 [lists.wikimedia.org](http://lists.wikimedia.org/).

Ces enregistrements permettent de désigner les serveurs vers lesquels on peut acheminer les courriels pour un domaine déterminé. Le nombre suivant le type d’enregistrement MX indique la priorité d’utilisation des serveurs, plus ce nombre est petit, plus le serveur est prioritaire.

Enregistrement SRV (Service Record), par exemple :

 _sip._tcp.wikimedia.org. 86400 IN SRV 0 33 5060 [serveursip1.wikimedia.org](http://serveursip1.wikimedia.org/).

Le nom de l’enregistrement est constitué de la concaténation des :

  • nom du service
  • protocole
  • domaine

Les valeurs suivant la déclaration représentent, dans l’ordre :

  • La priorité du serveur
  • Le Poids du serveur (répartion de charge à priorité égale)
  • Le port réseau utilisé par le service
  • La cible (adresse IP ou nom canonique)

Au final, un grand nombre d’autres types d’enregistrements sont également disponibles pour la définition d’une zone. (cf. https://fr.wikipedia.org/wiki/Liste_des_enregistrements_)

Parmi ceux-ci, les plus utilisés

  • TXT : offre la possibilité d’enregistrer un texte libre. (par exemple, les enregistrements SPF sont définis sous forme d’un enregistrement TXT)
  • LOC : définit une position géographique pour un objet de la zone.
  • KEY, CERT : permettent de fournir des clés ou des certificats cryptographiques associés au domaine.
  • SIG, KEY, DS, RRSIG, NSEC, DNSKEY : Fonctionnement en mode DNSSEC qui permet de sécuriser les enregistrements via le mécanisme de signature cryptographique des enregistrements.

Utilisation basique : Dans cette forme, dig va interroger le serveur DNS par défaut défini dans le fichier /etc/resolv.conf. Protocole réseau applicatif : HTTP

Les protocoles réseau pour l’accès au web :

HTTP : acronyme pour Hyper Text transfert Protocol, est un protocole textuel, toutes les informations circulent en clair (y compris les mots de passe). Différentes version successives :

HTTP 1.0 : 1 IP => 1 site servi ! HTTP 1.1 : 1 IP => Plusieurs sites possibles (notion d’hôtes virtuels) HTTP 2.0 : Objectif de performances avec l’établissement d’un mode connecté. HTTPS : En fait, HTTP encapsulé dans une connexion SSL (Secure Socket Layer). Toute la communication est cryptée, y compris la requête initiale de la version 1.1 :

GET / HTTP/1.1
Host: [www.example.com](http://www.example.com/)

qui indique vers quel hôte virtuel la requête est dirigée ! En conséquence, le serveur Web et les navigateurs récents doivent également implémenter le protocole SNI (Server Name Indication) afin de déterminer le certificat cryptographique adapté (en ce qui concerne Apache la fonctionnalité n’a été implémentée qu’à compter de la version 2.4).

Les ports par défauts réservés par l’IANA sont le 80/TCP pour HTTP et 443/TCP pour HTTPS.

Une fois la communication établie avec le serveur, le protocole définit différents méthodes permettant de dialoguer : GET : Demande d’une ressource HEAD : Demande d’informations sur une ressource POST : Soumission de données à une ressource vue d’un traitement (formulaire, envoi de ressources, modification de ressources, etc …) OPTIONS : Obtention des options de communication pour une ressource particulière ou le serveur en généralement. CONNECT : Etablissement d’un tunnel de communication vers un proxy HTTP. PUT : Envoi d’un ressource vers le serveur. PATCH : Modification d’une ressource sur le serveur. DELETE : Suppression d’une ressource sur le serveur. TRACE : Méthode de test de la connexion.

Serveur Web : APACHE2

Apache est actuellement l’un des serveurs web les plus répandus mais n’est pas le seul (IIS sous Windows, nginx sous les OS Unix-like, etc …).

Il est modulaire (dans le sens qu’il est possible d’activer ou désactiver des fonctionnalités selon les besoins) et hautement configurable ; d’ailleurs les responsables du paquet Debian ont découpé le fichier de configuration en nombre de fichiers plus spécialisés inclus dans le fichier principal, et fournissent des commandes spécialisées pour gérer l’activation des modules, des configurations et des sites virtuels.

L’empaquetage Debian du serveur Apache ajoute des commandes particulières pour gérer sa configuration :

a2enmod et a2dismod pour activer ou désactiver un module.

Ex :

a2enmod ssl, va activer le module SSL d’Apache.

a2enconf et a2disconf pour aectiver ou désactiver un fragment de configuration.

Ex :

a2enconf phpmyadmin #activera la configuration du service phpMyAdmin.

a2ensite et a2dissite pour activer ou désactiver un site virtuel (ou vhost dans la terminologie d’Apache)

Ex :

a2ensite default-ssl #activera le vhost SSL par défaut.

Les commandes a2(en|dis)(mod|conf|site) fonctionnent suivant la même politique d’implémentation :

Chaque fragment de configuration disponible se trouve dans un répertoire (modules|conf|sites)-available et l’activation ou la désactivation s’effectue en créant un lien symbolique dans le répertoire (modules|conf|sites)-enabled. Seuls les fichiers (donc les liens) se trouvant dans les répertoires (modules|conf|sites)-enabled font l’objet d’une directive include depuis le fichier de configuration principal : /etc/apache2.conf

Par exemple, la commande

a2ensite default-ssl

va procéder à la création d’un lien symbolique du fichier /etc/apache2/sites-available/default-ssl.conf dans le répertoire /etc/apache2/sites-enabled/ ;=> au rechargement du service ce fichier sera donc inclus depuis le fichier de configuration principal et le vhost correspondant se trouvera donc activé. L’implémentation du protocole HTTP 1.1 qui permet de servir différents sites web depuis un même serveur (en fait, une même adresse IP) a introduit la notion de site virtuel (vhost). La configuration d’un vhost minimal doit comporter la directive DocumentRoot qui définit la racine (le répertoire de base des fichiers servis par le site). Par exemple : DocumentRoot /var/www/html définit ce répertoire comme racine du site, et la requête [http://monsite/index.html](http://monsite/index.html) ciblera donc le fichier index.html se trouvant dans le répertoire /var/www/html/ Lorsque le fichier de configuration d’un vhost comporte les directives NameServer ou ServerAlias, apache2 ne sert ce fragment de configuration que si la requête ne cible que le nom d’hôte (ou de domaine) défini dans ces directives. Par exemple, un vhost configuré avec les directives Nameserver monsite.com et ServerAlias [www.monsite.com](http://www.monsite.com/) ne sera interrogé que si la requête cible : [http://monsite.com/](http://monsite.com/) ou http://www.monsite.com Par défaut, l’installation du paquet apache2 fournit 2 fichiers de configuration 000-default.conf et default-ssl.conf qui ne comportent pas de directive NameServer et dont la racine est déclarée (par la directive DocumentRoot) sur le dossier /var/ww/html/ Ces fichiers déterminent donc le vhost par défaut servi par apache2 sur la machine qui servira toute requête aboutissant sur la machine ne ciblant pas un nom d’hôte figurant dans une directive NameServer ou ServerAlias de la configuration d’un autre vhost.

Par exemple, si l’on rajoute une nouvelle configuration (monsite.conf) pour un vhost ciblant monsite.com avec la directive Nameserver monsite.com : http://monsite.com/ → servira les fichiers se trouvant dans le DocumentRoot du nouveau vhost défini par le fragment de configuration monsite.conf http://IP_du_serveur/ → servira les fichiers se trouvant dans /var/www/html/ comme défini par la définition du vhost par défaut (configuration du frangment 000-default.conf) Ce comportement reste identique en cas d’accès via le protocole HTTPS.

Protocole réseau applicatif : SMTP

Le protocole SMTP (pour Simple Mail Transfert Protocol) a été défini afin de transférer des courriers électronique vers des serveurs de messagerie électronique. Il s’agit également d’un protocole textuel (la norme initiale prévoit uniquement l’emploi d’un alphabet encodé en ASCII sur 7 bits) permettant de définir une méthode de dialogue entre client (MUA – Mail User Agent) et le serveur (MTA – Mail Transfert Agent).

L’IANA a réservé pour ce protocole différents ports :

  • 25/TCP pour la communication sans chiffrement.
  • 465/TCP pour une communication avec chiffrement explicite (encapsulation dans la couche TLS).
  • 587/TCP (aussi appelé port Submission) pour une communication chiffrée implicite et originellement destinée au dialogue entre MUA et MTA, afin d’assurer notamment des méthodes d’authentification.

MSA : Message Submission Agent (logiciel chargé de transmettre le message) MDA : Mail Delivery Agent (logiciel chargé de stocker le message) Protocole réseau applicatif : SMTP

Serveur Mail : Postfix

Différentes implémentations logicielles libres pour le rôle de MTA ont vu le jour, depuis l’ancestral sendmail. Actuellement, l’un des plus déployés sur GNU/Linux est Postfix, bien qu’il existe d’autres alternatives telles que Exim ou encore Qmail. Postfix a été initialement développé en 1999 par Wietse Venema pour IBM et distribué sous licence libre IBM Public Licence. Son succès provient essentiellement de son architecture modulaire, multi-processus avec gestion de files d’attente, ainsi que de sa relative facilité de configuration.

La configuration du paquetage Debian s’effectue principalement dans 2 fichiers situées dans le répertoire /etc/postfix/[main.cf](http://main.cf/) : c’est le fichier de configuration principal, il permet de référencer des fichiers complémentaires, et notamment des listes (alias, sites virtuels, utilisateurs virtuels, listes blanches ou noires, etc…). master.cf : Ce fichier permet de gérer la configuration des connexions réseau, ainsi que le chaînage avec différents processus optionnels et complémentaires (grey-listing, vérification spams et antivirus, etc …)

Par soucis d’efficacité, les fichiers comportant des listes, sont généralement transformés en BDD au format Berkeley DB via l’utillitaire postmap. Dans ce cas de figure, toute modification d’une liste implique la régénération de fichier .db correspondant et le rechargement de la configuration de Postfix. L’une de ces listes les plus utilisées, le fichier aliases (qui définit la liste des alias de messagerie du système) dispose d’un outil dédié : newaliases, permettant la prise en compte de ces modifications.

Protocole réseau applicatif : POP3

POP3 (Post Office Protocol) est un protocole textuel qui permet de récupérer les courriers électroniques stockés sur le serveur jouant le rôle de MTA. Le port par défaut réservé par l’IANA est le 110/TCP pour des transferts en clair ; une version chiffrée par encapsulement TLS est également définie (POP3S) avec pour port par défaut en 995/TCP. Dans la perspective de ne proposer que POP3 comme protocole de relève, le MDA du serveur peut utiliser le format mailbox pour stocker les messages. A la différence de l’autre protocole de relève de mails (IMAP), POP3 permet de conserver les messages sur le serveur de messagerie ; il suffit dans ce cas de paramétrer le client pour ne pas demander la suppression du message après la récupération (comportement par défaut des clients). Il existe différentes implémentations de serveur POP3 sous linux, allant du simplissime pop3d aux suites logicielles courier, cyrus ou dovecot.

Protocole réseau applicatif : IMAP

IMAP (Internet Message Access Protocol) est un second protocole textuel permettant de récupérer les courriers électroniques stockés sur le serveur jouant le rôle de MTA. Le port par défaut réservé par l’IANA est le 143/TCP pour des transferts en clair ; une version chiffrée par encapsulement TLS est également définie (IMAPS) avec pour port par défaut en 993/TCP. A la différence de l’autre protocole de relève de mails (POP3), IMAP prévoit de conserver les messages sur le serveur de messagerie ; Normalement, le client interroge le serveur tant pour obtenir la liste des messages que pour afficher un message particulier. De ce fait, le MDA configuré sur le serveur devra impérativement utiliser le format de stockage maildir pour enregistrer les mails. Il existe différentes implémentations du protocole IMAP sous GNU/Linux, allant du simplissime imapd aux suites logicielles courier, cyrus ou dovecot.

PROTOCOLE APPLICATIF : SNMP

Le protocole SNMP (pour Simple Network Management) a été prévu afin de permettre aux administrateurs de superviser, diagnostiquer et gérer à distance les équipements du réseau. 3 éléments principaux sont définis par le protocole :

  • Le manager est une console d’administrateur.
  • Les nœuds, sont les équipements à surveiller.
  • Les agents, sont des composants logiciels permettant de transmettre l’information depuis les nœuds.

Du point de vue applicatif, SNMP fait transiter ses informations via des datagrammes UDP (le port par défaut alloué par l’IANA est udp/161). C’est généralement le manager qui interroge périodiquement les agents afin de récupérer les informations et les métriques des matériels supervisés (polling). Outre ces datagrammes, les agents peuvent également générer des interruptions (traps) qui transitent également via des datagrammes UDP mais sur le port udp/162. Ces traps correspondent généralement à des dépassements de seuils ou encore au basculement d’un état.

Du point de vue du protocole, les matériels (commutateurs, concentrateurs, routeurs, postes de travail ou serveurs) contiennent des objets gérables.

Ces objets peuvent être des informations (status), des paramètres de configuration ou encore des statistiques de fonctionnement (métriques). Ils sont classés dans une sorte de base de données arborescente définie par l’ISO dénommée MIB (Management Information Base) La structure de la MIB est hiérarchique, et à chaque information décrite (object) correspond un identifiant d’objet (object identifier) qui permet de la cibler de façon inéquivoque. Ces identifiants sont représentés par une suite de chiffres séparés par des points. par exemple :

L’oi 1.3.6.1.2.1.2.2.1.2 cible l’objet ifDescr, défini comme étant une chaîne de caractères décrivant une interface réseau.

Les MIBs sont décrites en utilisation la notation ASN.1, ce qui donne par exemple :

`

escr OBJECT-TYPE

SYNTAX DisplayString (SIZE (0..255))
ACCESS read-only
STATUS mandatory
DESCRIPTION

"A textual string containing information

about the

interface. This string should include the

name of

the manufacturer, the product name and

the version

of the hardware interface."
::= { ifEntry 2 }

Pour la définition de l’oi ifDescr précédent. Plusieurs versions du protocole SNMP ont vu le jour au fur et à mesure des besoins de sécurité (actuellement v3). Le schéma de sécurité des versions 1 et 2 est uniquement basé sur la notion de communauté, qui est utilisé comme un mot de passe, et transite en clair sur le réseau. Généralement, les communautés par défaut sur les équipements se résument aux valeurs public ou private. De plus, à une valeur de communauté, le protocole associe également la notion d’accès en écriture ou lecture seule.

La version 3 a introduit un chiffrement cryptographique DES avec deux mots de passe (clés 64 bits) partagés entre l’agent et le manager (l’un pour l’authentification réciproque et l’autre servant à chiffrer le trafic).

PROTOCOLE APPLICATIF : LDAP

Le protocole LDAP (Lightweight Directory Access Protocol) permet d’accéder à des bases d’informations sur les utilisateurs d’un réseau, via l’interrogation d’annuaires. Il utilise pour cela le protocole TCP/IP (port 389) situé plus bas dans les couches OSI. Développé au début des années 1990, le protocole LDAP était destiné à remplacer le protocole DAP, utilisé pour accéder aux services d’annuaires X.500 de l’Open System Interconnection. Il s’est depuis transformé en annuaire natif pour pouvoir accéder à d’autres types d’annuaires. Normalisé par l’IETF (Internet Engineering Task Force), le protocole LDAP livre aux utilisateurs des méthodes qui leur permettent, entre autres, de se connecter et se déconnecter d’un réseau, d’insérer, modifier ou supprimer des entrées, de rechercher et de comparer des informations. Il dispose de mécanismes de chiffrement afin de sécuriser l’accès aux informations. La version actuelle de la norme est LDAP v3 ; et il a également été définie une version sécurisée via l’encapsulation TLS : LDAPS (port standard TCP/636).

Le but final du protocole LDAP étant de permettre l’interrogation et la gestion d’un annuaire, il prévoit un certain nombre d’opérations classiques :

  • Bind : indique la version utilisée et permet l’authentification du client.
  • Search : recherche dans l’annuaire et récupération des données.
  • Compare : test si une entrée contient un attribut avec une valeur déterminée.
  • Add : ajout d’une entrée (objet) dans l’annuaire.
  • Delete : suppression d’une entrée dans l’annuaire.
  • Modify : modification d’une entrée dans l’annuaire.
  • Modify DN : déplacement ou renommage d’une entrée dans l’annuaire.
  • Abandon : annulation d’un précédente requête.
  • Extended Operation : permet de définir d’autres opérations.
  • Unbind : fermeture de la connexion.

La norme LDAP prévoit également la possibilité de synchronisation entre serveurs afin de permettre la redondance des données et une mécanique de répartition de charge. Les annuaires LDAP sont structurés suivant un modèle répondant à la norme X.500, avece organisation hiérarchique et une architecture multi-tenant. Un annuaire est un arbre d’entrées (plusieurs racines sont possibles). Une entrée est constituée d’un ensemble d’attributs. Un attribut possède un nom, un type et une ou plusieurs valeurs (c’est ce qui diffère les annuaires LDAP d’une BDDR!), et s’il n’a pas de valeur l’attribut est considéré comme absent de l’annuaire.

Les attributs sont définis dans des schémas.

Les entrées d’un annuaire LDAP sont définies par des schémas qui déterminent les types d’attributs et donc la validité de leur contenu. Les schémas définissent des classes d’objets, et chaque entrée dans l’annuaire doit au moins contenir une valeur pour l’attribut objectClass afin de permettre de valider son contenu.

Il est donc possible d’étendre le contenu d’un annuaire LDAP en chargeant des schémas supplémentaires. Ainsi, un annuaire destiné à identifier des utilisateurs Unix ne devra disposer des mêmes schémas qu’un annuaire Active Directory ; et, de même l’installation d’un service Share Point rajoutera de nouveaux schémas dans l’Active Directory. PROTOCOLE APPLICATIF : LDAP

Exemples d’attributs courants :

cn : pour Common Name, le nom d’usage de l’élément. dc : pour Domain Component, représente une partie de nom de domaine (DNS). objectClass : le schéma décrivant l’élément courant. ou : pour Organizational Unit, désigne généralement un groupe. uid : identifiant unique pour un utilisateur Unix. gid : identifiant unique pour un groupe Unix.

D’une manière générale, la racine d’un annuaire est calquée sur l’organisation du DNS, mais des bases différenciées peuvent également exister (par exemple, la configuration d’un serveur OpenLDAP est dorénavant stockée dans l’annuaire lui-même à partir de la racine dn:cn=config). Chaque entrée dans l’annuaire dispose d’un identifiant unique : le Distinguished Name (abrégé en DN) qui est constitué de son Relative Distingueshed Name (RDN) suivi du DN de son parent. Le DN représente un chemin absolu vers un élément alors que le RDN serait un chemin relatif permettant d’identifier un élément parmi les autres au sein d’une même structure de l’arborescence. Ex : rdn: uid=1000 ou dn: uid=1000,dc=adm,dc=lan

Un format particulier a été établi pour représenter textuellement les entrées d’un annuaire LDAP : le format LDIF (pour LDAP Data Interchange Format).

Exemple :

dn: dc=example,dc=com
dc: example
description: Serveur exemple
objectClass: dcObject
objectClass: organization
o: Serveur exemple
dn: ou=people, dc=example,dc=com
ou: people
objectClass: organizationalUnit
